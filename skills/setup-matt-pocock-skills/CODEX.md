# Setup Matt Pocock's Skills — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `setup-matt-pocock-skills`; логика и процесс те же, что в [SKILL.md](SKILL.md) (explore → present three decisions по одной → confirm draft → write `## Agent skills` блок в CLAUDE.md/AGENTS.md + три файла `docs/agents/*.md`). Здесь только дельта под Codex.

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md` (корень проекта или `~/.codex/AGENTS.md`). Авто-триггера по полю `description`, как у Claude-скиллов, у Codex нет — фронтматтер из SKILL.md (включая `disable-model-invocation: true`) Codex не интерпретирует. Активировать можно тремя способами:

1. **Кастомный промпт для слэш-вызова.** Положи тело SKILL.md (без YAML-фронтматтера) в `~/.codex/prompts/setup-matt-pocock-skills.md`. Тогда в сессии вызывается `/setup-matt-pocock-skills` — это эквивалент `/`-команды Claude Code.
2. **Правило в AGENTS.md.** Добавь блок ниже, чтобы Codex знал, когда и как запускать setup.
3. **Вручную.** Просто скажи Codex: «прогони setup-matt-pocock-skills» и дай ссылку на SKILL.md.

Этот скилл намеренно не самозапускающийся (`disable-model-invocation: true` в оригинале) — его запускают явно, один раз на репозиторий перед первым использованием инженерных скиллов. Поэтому вариант 1 или 3 предпочтительнее авто-правила.

Минимальный блок для `AGENTS.md`:

```md
## Setup инженерных скиллов (setup-matt-pocock-skills)
Запускается ЯВНО (по команде пользователя или /setup-matt-pocock-skills), один раз
на репозиторий. Цель: создать `## Agent skills` блок в CLAUDE.md/AGENTS.md и файлы
`docs/agents/issue-tracker.md`, `triage-labels.md`, `domain.md`, чтобы скиллы to-issues,
to-prd, triage, diagnose, tdd, improve-codebase-architecture, zoom-out знали трекер
задач, словарь triage-меток и раскладку доменных доков.
Процесс: explore репо → три решения ПО ОДНОМУ (трекер / метки / домен) → показать
черновик → записать. Сам не инициируй — жди явного запроса.
Полная версия — skills/setup-matt-pocock-skills/CODEX.md и SKILL.md.
```

## Что меняется против Claude-версии

Скилл прост по механике: один прогон, интерактивный диалог, в конце пишет файлы. Субагентов, harness-worktree-изоляции и bundled-скриптов на запуск тут нет — соответствующих дельт не возникает. Реально применимы только две.

### 1. CLI трекера задач: `gh` / `glab` — пользователь ставит и авторизует сам

Скилл при выборе трекера прописывает в `docs/agents/issue-tracker.md` команды `gh issue ...` (GitHub) или `glab issue ...` (GitLab). У Codex нет встроенного GitHub-инструмента — это обычные внешние CLI:

- На фазе **explore** Codex выполняет `git remote -v` обычным shell-вызовом (это есть у Codex из коробки) — чтобы предложить трекер по умолчанию.
- Сами `gh` / `glab` должны быть **установлены и авторизованы пользователем** (`gh auth login` / `glab auth login`). Setup только записывает шаблон команд в `docs/agents/issue-tracker.md`; фактически их потом вызывают инженерные скиллы.
- Если предполагается дёргать трекер через MCP, а не через CLI — это настраивается пользователем в `~/.codex/config.toml` (секция MCP-серверов). Но дефолт скилла — именно CLI, и он переносится 1-в-1.
- **Guard:** если на этапе explore `gh`/`glab` отсутствует или не авторизован — не выдумывай вывод и не подставляй фиктивный remote. Сообщи пользователю и предложи local-markdown (`.scratch/`) как трекер без зависимости от CLI.

### 2. Слэш-вызов и «свежий прогон»

`/setup-...` в Claude Code → в Codex это промпт-файл `~/.codex/prompts/setup-matt-pocock-skills.md` (см. выше). Скилл одношаговый, контекст между шагами копить не нужно; если хочется воспроизводимого неинтерактивного прогона (например, в скрипте инициализации репо) — `codex exec "прогони setup-matt-pocock-skills по skills/setup-matt-pocock-skills/SKILL.md"`. Но по своей природе скилл диалоговый (три подтверждения подряд), поэтому интерактивная сессия удобнее, чем `exec`.

### GPT-специфика

- Держи три решения **строго по одному** (трекер → метки → домен), как требует SKILL.md, — не вываливай все сразу. GPT-модели склонны схлопывать диалог; следи за этим явно.
- Перед записью **покажи черновик** блока `## Agent skills` и трёх файлов и дождись подтверждения — это гейт из шага 3, не пропускай его.
- Правила выбора целевого файла соблюдай буквально: есть `CLAUDE.md` → правишь его; иначе `AGENTS.md`; нет ни одного → **спроси**, не выбирай сам. Никогда не создавай второй файл при наличии первого. Существующий блок `## Agent skills` — обновляешь in-place, не дублируешь.

## Ресурсные файлы — переносятся как есть

Seed-шаблоны в папке скилла читаются Codex'ом с диска так же, как Claude'ом, и копируются/адаптируются под выбор пользователя без изменений:

- [issue-tracker-github.md](issue-tracker-github.md), [issue-tracker-gitlab.md](issue-tracker-gitlab.md), [issue-tracker-local.md](issue-tracker-local.md) — основа для `docs/agents/issue-tracker.md` (для «Other»-трекера файл пишется с нуля по описанию пользователя);
- [triage-labels.md](triage-labels.md) — маппинг пяти канонических ролей на реальные метки;
- [domain.md](domain.md) — правила чтения `CONTEXT.md` / `CONTEXT-MAP.md` / `docs/adr/`.

Это не исполняемые скрипты, а текстовые заготовки — авто-запуска у Codex и не требуется.
