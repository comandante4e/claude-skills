# Prototype — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `prototype`; вся логика и процесс — те же, что в [SKILL.md](SKILL.md) (и в ветках [LOGIC.md](LOGIC.md), [UI.md](UI.md)). Здесь только дельта механики под Codex.

Скилл почти чисто инструкционный: это набор правил «как и зачем строить одноразовый прототип», без субагентов, без bundled-скриптов и без MCP-инструментов. Поэтому переносится в Codex практически 1-в-1 — меняется в основном то, **где он живёт** и **как триггерится**, плюс пара мелочей про запуск.

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md`. Положи правило в:
- `AGENTS.md` в корне проекта (или нужного пакета), **либо**
- `~/.codex/AGENTS.md` — глобально на все проекты.

Авто-триггера по `description` (как у Claude-скиллов) у Codex **нет**. Варианты активации этого скилла:
- **Правило в `AGENTS.md`** — Codex увидит его и пойдёт по веткам ниже, когда пользователь скажет «прототип / прощупать / mock up / try a few designs».
- **Кастомный промпт `~/.codex/prompts/prototype.md`** — даёт слэш-вызов `/prototype` в чате Codex. Содержимое промпта — текст SKILL.md (или ссылка-инструкция «следуй skills/prototype/SKILL.md»).
- **Вручную** — просто открыть SKILL.md и вести работу по нему.

Минимальный блок для `AGENTS.md`:

```md
## Прототип (одноразовый код, отвечающий на вопрос)
Когда пользователь просит «прототип / прощупать дизайн / mock up / try a few designs»,
сначала выбери ветку:
- вопрос про логику/состояние/модель данных → skills/prototype/LOGIC.md
  (крошечный интерактивный TUI поверх чистого, переносимого модуля логики);
- вопрос «как это должно выглядеть» → skills/prototype/UI.md
  (несколько радикально разных UI-вариантов на одном роуте, переключаемых через ?variant=).
Общие правила (SKILL.md): throwaway с первого дня и помечен как таковой; одна команда запуска
через существующий таск-раннер проекта; без персистентности по умолчанию; без тестов/абстракций;
показывай полное состояние после каждого действия; по итогу — зафиксируй ОТВЕТ и удали/влей прототип.
Полная версия — skills/prototype/CODEX.md.
```

## Что меняется против Claude-версии

Скилл не опирается на спавн субагентов, `git worktree`-изоляцию харнесса, bundled-скрипты или конкретные MCP-инструменты — поэтому большинство «тяжёлых» дельт неприменимы. Реальные отличия — короткие.

### 1. Слэш-вызов `/prototype`
В Claude Code скилл триггерится по `description`. В Codex эквивалент `/prototype` — это кастомный промпт-файл `~/.codex/prompts/prototype.md` (см. выше) либо правило в `AGENTS.md`. Без одного из них Codex про скилл не знает — надо указать на него явно.

### 2. «Одна команда запуска» — проверь руками, не полагайся на авто-обнаружение
Оба правила («одна команда запуска» в SKILL.md, шаг 5 в LOGIC.md) требуют добавить скрипт в существующий таск-раннер (`package.json` scripts, `Makefile`, `justfile`, `pyproject.toml`). У Codex нет авто-обнаружения раннера — **открой конфиг проекта и убедись**, какой раннер реально есть, прежде чем дописывать скрипт. Нет раннера — положи команду в начало README прототипа (как и сказано в LOGIC.md).

### 3. Прогон прототипа — через обычный shell, `codex exec` для воспроизводимости
- **Logic-ветка (TUI):** прототип интерактивный — его драйвит **пользователь** в своём терминале (`pnpm run <name>`, `python <path>`, `just <name>` и т.п.). Codex сам по себе интерактивный stdin TUI не «прокликает»; твоя задача — собрать код и **отдать команду запуска**, дальше человек жмёт клавиши. Это совпадает с шагом 6 LOGIC.md «Hand it over».
- **UI-ветка:** поднять дев-сервер и выдать URL с ключами `?variant=`. Запуск самого дев-сервера, если нужен воспроизводимый/скриптовый прогон, можно обернуть в `codex exec "<команда запуска>"`; интерактивную правку вариантов веди в обычной сессии.
- Флаги вроде «спрятать switcher в проде» (`NODE_ENV !== 'production'`) ставит сам код прототипа — Codex тут ничего не делает по-особому.

### 4. Guard: нет раннера/команды — не выдумывай
Если в проекте нет очевидного раннера или команды запуска — **не изобретай новый top-level раннер/структуру** (это и так запрещено правилом 1 SKILL.md и шагом 5 LOGIC.md). Остановись, сообщи пользователю и предложи положить команду запуска в README прототипа.

### 5. GPT-специфика
- Первое и **самое дорогое** решение — выбор ветки (LOGIC vs UI). Прими его явно, по чек-листу из SKILL.md (бэкенд-модуль → logic; страница/компонент → UI); при неоднозначности и недоступном пользователе — задефолться по окружающему коду и **зафиксируй допущение в шапке прототипа**.
- Не перепрыгивай шаги веток: для LOGIC — сначала «сформулировать вопрос» → выделить чистый модуль → тонкий TUI-шелл; для UI — сначала N и «вопрос» → радикально разные варианты → switcher.
- По завершении не забудь шаг «Capture the answer»: ответ на вопрос (commit message / ADR / issue / `NOTES.md` рядом) — единственное, что стоит сохранить из прототипа.

## Ресурсные файлы

[LOGIC.md](LOGIC.md) и [UI.md](UI.md) — две ветки скилла; переносятся **как есть** и читаются Codex'ом с диска так же, как Claude. Никакой адаптации их содержимого не требуется — вся механика в них (чистый модуль логики + TUI; `?variant=` + floating switcher) платформо-независима.
