# To Issues — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `to-issues`; логика и процесс те же, что в [SKILL.md](SKILL.md) (контекст → исследование кода → вертикальные срезы-«трассеры» → квиз с пользователем → публикация issue по шаблону). Здесь только дельта под Codex.

Скилл почти чисто-инструкционный: вся его суть — текстовый процесс разбиения плана на тонкие end-to-end срезы и шаблон тела issue. Под Codex меняются ровно две вещи: **где он живёт / как триггерится** и **чем именно публикуются issue** (у Claude это был встроенный issue-tracker-инструмент, у Codex его надо подключить или вызвать руками).

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md` (корень проекта или `~/.codex/AGENTS.md`). Авто-триггера по `description` (как у Claude-скиллов) у Codex **нет** — фразу «разбей план на issue / создай тикеты» должен ловить либо загруженный `AGENTS.md`-блок, либо ты вызываешь скилл вручную.

Три способа активации (выбери один):

1. **Кастомный промпт для слэш-вызова** — положи тело SKILL.md (процесс + шаблон issue) в `~/.codex/prompts/to-issues.md`. Тогда в чате доступен вызов `/to-issues` (можно с аргументом — ссылкой/номером/путём существующего issue, как в шаге 1 процесса).
2. **Правило в `AGENTS.md`** — короткий блок ниже, чтобы Codex шёл по воркфлоу по ключевой фразе.
3. **Вручную** — просто скорми SKILL.md как инструкцию в начале сессии.

Минимальный блок для `AGENTS.md`:

```md
## Разбивка плана на issue (to-issues)
Когда пользователь просит разбить план/spec/PRD на independently-grabbable issue
(или передаёт ссылку/номер существующего issue) — веди процесс по skills/to-issues/SKILL.md:
вертикальные срезы-трассеры (тонкий end-to-end путь через ВСЕ слои), пометка HITL/AFK,
квиз пользователю до апрува, потом публикация по шаблону issue в порядке зависимостей.
Issue-трекер и словарь triage-меток должны быть заданы заранее; если их нет — спроси, не выдумывай.
Публикуй через подключённый инструмент трекера (MCP/CLI); нет инструмента — выведи готовый
markdown по шаблону и попроси пользователя создать issue, либо вызови CLI явно.
Не закрывай и не меняй родительский issue.
```

## Что меняется против Claude-версии

### 1. Слэш-вызов `/to-issues`
`/to-issues` в Claude Code → в Codex это кастомный промпт `~/.codex/prompts/to-issues.md` или правило в `AGENTS.md` (см. выше). Аргумент (номер/URL/путь существующего issue из шага 1) передаётся так же — после имени промпта.

### 2. Issue-трекер — это внешний инструмент, его надо подключить
Шаги «fetch из трекера» (шаг 1) и «publish to the issue tracker» (шаг 5) опираются на доступ к трекеру. У Codex встроенного трекер-инструмента нет — подключи один из вариантов:
- **MCP-сервер трекера** (GitHub/Linear/Notion и т.п.) в `~/.codex/config.toml` — тогда fetch и публикация идут через его инструменты;
- **CLI** — для GitHub это `gh issue create`/`gh issue view` (вызывается из шелла), для других трекеров — их CLI/`curl` к API;
- если ничего не подключено — Codex **не имеет** способа создать issue сам: выведи готовый markdown по `<issue-template>` для каждого среза (в порядке зависимостей) и попроси пользователя запостить, либо собери команды `gh issue create ...` и выполни их явно.

«Issue tracker и словарь triage-меток должны быть заданы заранее» (преамбула SKILL.md и ссылка на `/setup-matt-pocock-skills`): этой setup-команды под Codex может не быть. **Нет заданного трекера/меток — остановись и спроси пользователя**, не подставляй трекер и метки наугад.

### 3. Domain glossary / ADR при исследовании кода
Шаг 2 (titles/descriptions из глоссария проекта, уважать ADR) переносится 1-в-1: Codex читает эти файлы с диска так же, как Claude. Просто убедись, что они доступны в дереве проекта.

### 4. GPT-специфика
- Гейт-квиз (шаг 4) — **не перепрыгивай**: сначала покажи нумерованный список срезов (Title / HITL-AFK / Blocked by / User stories), дождись явного апрува гранулярности и зависимостей, и только потом публикуй.
- Публикуй в порядке зависимостей (блокеры первыми), чтобы в поле «Blocked by» стояли реальные идентификаторы созданных issue — под Codex это особенно важно делать пошагово, без забегания вперёд.
- Для воспроизводимой пакетной публикации (скрипт/CI) используй `codex exec "<промпт + контекст плана>"` неинтерактивно; исследование кода и квиз — в интерактивной сессии.

## Чего у этого скилла НЕТ (и не надо добавлять)
Скилл не спавнит субагентов, не использует git worktree и не несёт bundled-скриптов (`scripts/`, `data/`) — в папке только `SKILL.md`. Поэтому секции про «свежий агент на шаг», параллельные деревья и явный запуск вспомогательных скриптов здесь неприменимы. Процесс и `<issue-template>` переносятся как есть; единственная реальная инфраструктурная зависимость — доступ к issue-трекеру (см. п.2).
