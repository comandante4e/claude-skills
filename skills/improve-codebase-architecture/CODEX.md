# Improve Codebase Architecture — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `improve-codebase-architecture`; логика и процесс — те же, что в [SKILL.md](SKILL.md) (исследовать → предложить кандидатов → grilling-цикл). Глоссарий, принципы и все ресурсные файлы переносятся как есть. Здесь только дельта механики под Codex.

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md`. Авто-триггера по `description` (как у Claude-скиллов) у Codex **нет** — активацию задаёшь сам одним из трёх способов:

1. **Правило в `AGENTS.md`** (корень проекта или `~/.codex/AGENTS.md` глобально) — Codex подхватит его и будет вести архитектурный разбор по процессу ниже.
2. **Кастомный промпт** `~/.codex/prompts/improve-codebase-architecture.md` — даёт слэш-вызов `/improve-codebase-architecture` в интерактивной сессии. В тело файла положи процесс из [SKILL.md](SKILL.md) и ссылки на ресурсные `.md`.
3. **Вручную** — открыть сессию и сказать «найди возможности углубления архитектуры в этом коде».

Минимальный блок для `AGENTS.md`:

```md
## Архитектурный разбор (deepening)
Когда пользователь просит улучшить архитектуру / найти рефакторинг / снизить связность /
сделать код тестируемее и навигабельнее для ИИ — веди разбор по skills/improve-codebase-architecture/.
Сначала прочитай CONTEXT.md (доменный глоссарий) и docs/adr/. Лексика: module, interface,
implementation, depth, seam, adapter, leverage, locality (см. LANGUAGE.md) — не подменяй на
component/service/API/boundary. Процесс: explore → пронумерованный список кандидатов (НЕ предлагать
интерфейсы сразу) → grilling-цикл по выбранному. Файлы LANGUAGE.md / DEEPENING.md / INTERFACE-DESIGN.md
читаются с диска так же.
```

## Что меняется против Claude-версии

### 1. Субагент Explore → одна сессия Codex (или `codex exec` на проход)
Шаг 1 «Explore» в SKILL.md вызывает Agent tool с `subagent_type=Explore` — у Codex встроенного спавна субагентов **нет**. Маппинг:
- обход кодовой базы делаешь **в текущей сессии** обычными чтениями (`rg`/grep, чтение файлов) — для этого скилла субагент был лишь способом получить свежий контекст под разведку, не отдельная машинерия;
- если кодовая база большая и хочется изолировать разведочный проход от истории — запусти его **отдельной сессией** или `codex exec "пройди по <путям>, отметь shallow-модули и места трения, верни список файлов и наблюдений"`, а результат прочитай с диска / из вывода;
- сам алгоритм разведки не меняется: читаешь `CONTEXT.md` и релевантные ADR первыми, дальше ищешь трение органически и прогоняешь deletion-тест.

### 2. Параллельные сабагенты для альтернативных интерфейсов → несколько прогонов Codex
[INTERFACE-DESIGN.md](INTERFACE-DESIGN.md) шаг 2 спавнит 3+ сабагента параллельно через Agent tool, по разному дизайн-ограничению на каждого («минимизируй интерфейс» / «максимизируй гибкость» / «оптимизируй частый кейс» / «ports & adapters»). У Codex это делается руками:
- сформируй один технический бриф (пути, детали связности, категория зависимости из [DEEPENING.md](DEEPENING.md), что прячется за seam, лексика из LANGUAGE.md + CONTEXT.md);
- запусти **N отдельных прогонов** `codex exec` с одним и тем же брифом, но разным дизайн-ограничением в промпте; каждый пишет свой вариант (интерфейс, пример использования, что скрыто за seam, стратегия адаптеров, trade-offs) в свой файл, например `design-1.md`, `design-2.md`, …;
- эти прогоны независимы и реально параллельны (запусти в фоне), потому что каждый пишет в свой файл — конфликта артефактов нет;
- затем в основной сессии прочитай все варианты с диска, представь их по очереди и дай свою рекомендацию (шаг 3 без изменений).

Если параллель не нужна — можно последовательно в одной сессии, прося по одному варианту за раз; смысл «Design It Twice» не страдает.

### 3. Ресурсные файлы и кросс-скилловые ссылки — без изменений
- [LANGUAGE.md](LANGUAGE.md), [DEEPENING.md](DEEPENING.md), [INTERFACE-DESIGN.md](INTERFACE-DESIGN.md) — переносятся 1-в-1, Codex читает их с диска так же.
- Ссылки на соседний скилл (`../grill-with-docs/CONTEXT-FORMAT.md`, `../grill-with-docs/ADR-FORMAT.md`) работают, **только если каталог `grill-with-docs/` лежит рядом**. Если ставишь скилл отдельно — скопируй эти два файла к себе или поправь пути. Нет файла формата — не выдумывай структуру `CONTEXT.md`/ADR, открой и прочитай эталон.
- Сайд-эффекты grilling-цикла (создать/дополнить `CONTEXT.md`, предложить ADR) — это обычные правки файлов на диске, никакого харнесса не требуют. `CONTEXT.md` создавай лениво, если его ещё нет.

### 4. GPT-специфика и guard
- Держи процесс **по шагам**: explore → кандидаты → grilling. Не перепрыгивай на предложение интерфейсов до явного выбора пользователя — это гейт из SKILL.md.
- Кандидатов выдавай **пронумерованным списком** (Files / Problem / Solution / Benefits) и заканчивай вопросом «какой разобрать?» — не сваливай сразу в дизайн.
- Лексику соблюдай дословно (module/seam/adapter/…); деление на component/service/API/boundary — ошибка, ровно как в Claude-версии.
- Для воспроизводимых разведочных и дизайн-прогонов используй `codex exec` (неинтерактивно); интерактив — для grilling-диалога с пользователем.
