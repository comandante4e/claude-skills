# Diagnose — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `diagnose`; логика и процесс те же, что в [SKILL.md](SKILL.md) (6 фаз: feedback loop → reproduce → hypothesise → instrument → fix+regression-test → cleanup/post-mortem). Здесь только дельта механики под Codex.

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md` (корень проекта/пакета или `~/.codex/AGENTS.md` глобально). Авто-триггера по `description` (как у Claude-скиллов) у Codex **нет** — фразы «diagnose this», «debug this», «оно падает/тормозит» сами по себе скилл не поднимут. Поэтому дисциплину подключают одним из трёх способов:

1. **Правило в `AGENTS.md`** — короткий блок (ниже), Codex видит его в каждой сессии этого проекта и идёт по фазам, когда пользователь сообщает о баге/регрессии.
2. **Кастомный промпт для слэш-вызова** — положи полный текст процесса в `~/.codex/prompts/diagnose.md`, тогда `/diagnose` в чате запускает воркфлоу явно.
3. **Вручную** — просто скажи Codex «веди по diagnose из skills/diagnose/SKILL.md».

Минимальный блок для `AGENTS.md`:

```md
## Diagnose (дисциплина для трудных багов)
Когда пользователь сообщает о баге, что-то «падает/throwing/failing» или о регрессии
производительности — веди диагностику по фазам:
1) построй быстрый детерминированный pass/fail feedback loop (это и есть скилл);
2) воспроизведи баг на нём; 3) выпиши 3–5 ранжированных falsifiable-гипотез и покажи
их пользователю; 4) инструментируй по одной переменной, теги логов [DEBUG-xxxx];
5) регресс-тест перед фиксом (если есть корректный seam) → фикс → перезапуск loop;
6) cleanup: убрать [DEBUG-*], удалить прототипы, post-mortem.
Не переходи к гипотезам без работающего loop — нет loop, остановись и попроси
артефакт/доступ. Полная версия — skills/diagnose/CODEX.md и SKILL.md.
```

## Что меняется против Claude-версии

Скилл почти целиком **инструкционный** (дисциплина фаз, чек-листы, ранжирование гипотез). Субагентов, worktree-изоляции и MCP-инструментов он не использует, поэтому большая часть процесса переносится 1-в-1. Реальные дельты под Codex:

### 1. Bundled-скрипт `scripts/hitl-loop.template.sh` не запускается сам

Phase 1, способ №10 (last resort) опирается на `scripts/hitl-loop.template.sh` — bash-драйвер human-in-the-loop. У Codex **нет авто-подхвата вспомогательных скриптов из папки скилла**. Поэтому агент/пользователь вызывает его **явно**:

```bash
# скопировать шаблон рядом, отредактировать шаги step/capture под конкретный баг,
# затем запустить в терминале (человек следует подсказкам, вывод KEY=VALUE летит назад агенту):
cp skills/diagnose/scripts/hitl-loop.template.sh ./hitl-loop.sh
$EDITOR ./hitl-loop.sh
bash ./hitl-loop.sh
```

Codex запускает его через свой обычный shell-вызов; в неинтерактивном/скриптовом режиме — `codex exec "bash ./hitl-loop.sh"`. Сам скрипт правится как есть (редактируешь блок между `--- edit below ---` и `--- edit above ---`). Это «подводный камень»: путь к шаблону Codex не угадает — указывай его явно.

### 2. Feedback loop через `codex exec` (воспроизводимость)

Главный артефакт скилла — быстрый детерминированный loop (тест / curl / CLI-снапшот / headless-браузер / `git bisect run` / property-loop). Под Codex держи запуск loop **воспроизводимым**: оборачивай прогон в `codex exec "<команда теста/харнесса>"` или просто вызывай его командой через shell. `git bisect run <скрипт>`, Playwright/Puppeteer, curl-скрипты — это обычные CLI-инструменты, Codex дёргает их сам, никакой Claude-специфики тут нет. Никакого спавна субагентов на шаг скилл не требует — каждая фаза идёт в той же сессии, опираясь на сигнал loop.

### 3. Handoff `/improve-codebase-architecture` → промпт-файл или ручной вызов

Phase 6 предлагает передать находку в скилл `/improve-codebase-architecture`. В Codex слэш-команд Claude нет: либо положи этот скилл как `~/.codex/prompts/improve-codebase-architecture.md` и зови `/improve-codebase-architecture`, либо передай находку явным промптом в новой сессии. Если такого промпта/скилла на машине нет — **не выдумывай его поведение**: просто сформулируй архитектурную рекомендацию текстом (нет корректного test-seam, спутанные callers, скрытая связность) и оставь пользователю.

### 4. Пользовательские чек-пойнты остаются текстовыми

Phase 3 требует показать ранжированный список гипотез пользователю до проверки. В Codex это обычный текстовый вывод в чат — никакого инструмента, всё работает как есть. Если пользователь AFK — продолжай по своему ранжированию (как в SKILL.md).

### 5. GPT-специфика

- Иди **строго по чек-листу фаз**, не перепрыгивай гейты: «не переходи к Phase 2 без loop, которому веришь», «не гипотезируй без репро».
- **Одна переменная за раз** в Phase 4; каждый probe — под конкретное предсказание из Phase 3.
- Теги `[DEBUG-xxxx]` обязательны — финальная уборка в Phase 6 это один `grep` по префиксу.
- **Guard:** если loop построить не удаётся (нет окружения/артефакта/доступа) — остановись и сообщи явно, перечисли что пробовал, попроси HAR/лог/трейс/доступ. Не имитируй наличие сигнала.

## Ресурсные файлы

`scripts/hitl-loop.template.sh` переносится **как есть** — Codex читает и запускает его так же, как любой файл с диска; меняется только то, что вызов ручной (см. дельту №1). Других ресурсных `.md`/`data/` у скилла нет.
