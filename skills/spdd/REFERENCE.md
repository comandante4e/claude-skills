# SPDD — справочник

## Шаблон REASONS Canvas

````markdown
# REASONS Canvas: <story-id> — <короткое название>

## R — Requirements
**Проблема:** [одно предложение — какую боль решаем]
**Стейкхолдер:** [кто страдает / кто получит выгоду]
**Критерии успеха:**
- [ ] Измеримый критерий 1 (с метрикой/порогом)
- [ ] Измеримый критерий 2
- [ ] Acceptance-сценарий: Given / When / Then

## E — Entities
[Доменные модели, их атрибуты, инварианты, отношения]

Пример:
- **Order**
  - id: UUID
  - status: Pending | Paid | Shipped | Delivered
  - items: List<OrderItem>
  - Инвариант: status переходит только вперёд по линии
- **OrderItem**
  - sku, quantity, price
  - Инвариант: quantity > 0

## A — Approach
- Стратегия высокого уровня (1–3 предложения)
- Выбранные паттерны / библиотеки
- **Альтернативы:** что рассмотрели и почему отвергли
- Trade-offs: что выигрываем, чем платим

## S — Structure
- В какие модули/слои вписывается
- Затрагиваемые компоненты (новые / изменяемые)
- Точки интеграции (API, события, БД)
- Текстовая схема или диаграмма

## O — Operations
Пошаговый план реализации. Уровень детализации — до сигнатур.

1. Создать `OrderRepository.save(order: Order): Result<OrderId>`
2. Добавить миграцию `2026_05_23_orders.sql` со схемой …
3. Эндпоинт `POST /orders` → валидация → save → 201 / 400
4. ...

## N — Norms
- Code style: ссылка на проектные конвенции
- Нейминг: …
- Логирование: формат, уровни
- Обработка ошибок: Result vs Exception
- Тесты: фреймворк, расположение, покрытие

## S — Safeguards
**Запрещено:**
- Прямые SQL-запросы в обход репозитория
- Изменение схемы существующих таблиц без миграции
- Доступ к модулю X из модуля Y

**Инварианты безопасности:**
- Все эндпоинты под `/api/admin/*` требуют роль admin
- PII не попадает в логи
````

## Детали каждого шага

### Шаг 1: Story

- Запросить большое требование от пользователя
- Разбить на 3–7 INVEST-историй:
  - **I**ndependent — каждая deployable отдельно
  - **N**egotiable — детали обсуждаемы
  - **V**aluable — ценность для пользователя/бизнеса
  - **E**stimable — можно оценить
  - **S**mall — влезает в спринт
  - **T**estable — есть критерий «готово»
- Приоритизировать совместно с пользователем
- Артефакт: `docs/spdd/<story-id>/story.md` (1–3 абзаца + acceptance criteria)

### Шаг 2: Analysis

- Извлечь доменные концепты из истории (существительные → entities, глаголы → operations)
- Сканировать существующий код через Grep/Glob: где уже есть похожее, что переиспользовать
- Идентифицировать риски: техдолг, конфликты с существующими модулями, performance, security
- Предложить 1–2 альтернативы дизайна с trade-offs
- Артефакт: `docs/spdd/<story-id>/analysis.md`

### Шаг 3: Context

- Сжатый документ — то, что агент должен знать ПЕРЕД генерацией
- Разделы:
  - Доменный словарь (термины и их значение в этом проекте)
  - Scope: что входит, что НЕ входит
  - Связи с существующими модулями (read / write / dependency)
  - Технические предпосылки (стек, версии)
- Артефакт: `docs/spdd/<story-id>/context.md`

### Шаг 4: REASONS Canvas

- Заполнить все 7 секций по шаблону выше
- Operations — самая детальная секция, до сигнатур методов
- **Quality gate:** показать пользователю на ревью ДО генерации
- Если пользователь правит — фиксируем правку в канвасе, не в коде
- Артефакт: `docs/spdd/<story-id>/reasons-canvas.md`

### Шаг 5: Generate

Порядок:
1. **Acceptance-тесты** из Requirements (Given/When/Then) — пишутся первыми, красные
2. **Код** из Operations — шаг за шагом, не перескакивая
3. Acceptance-тесты позеленели → переход к шагу 6

При логической ошибке во время генерации:
- Ошибка в требованиях → правь канвас → перегенерируй затронутые секции
- Код «уехал» в рефакторинге → синхронизируй канвас с кодом (sync обратной стороной)
- НЕ молча менять реализацию без обновления канваса

### Шаг 6: Unit Tests

- Покрытие edge cases из Entities (инварианты) и Safeguards (запреты)
- Не дублировать acceptance-тесты из шага 5
- Фокус: одна единица — один тест-кейс

## Признаки рассинхронизации (стоп-сигналы)

Если видишь хотя бы один — остановись и обсуди с пользователем:
- Реализация противоречит Safeguards
- Логика не покрыта в Operations
- Появилась новая entity, не описанная в Entities
- Tests passing, но критерий из Requirements не выполнен
- Пользователь просит «просто поправь код» — напомни про правило «промпт первым»

## Когда НЕ применять (повтор)

- **Hotfix** продакшен-инцидента — нужна скорость, не процесс
- **Спайк/прототип-разведка** — REASONS преждевременен, домен ещё не ясен. Используй скилл `prototype`
- **Тривиальное изменение** — одна строка, переименование, опечатка
- **Чистая исследовательская работа** — `improve-codebase-architecture` или ручной flow

В этих случаях — обычный flow без скилла.

## Связь с другими скиллами

- `tdd` — шаг 5 SPDD по сути TDD red-green-refactor, можно делегировать
- `grill-me` / `grill-with-docs` — для жёсткого ревью канваса на шаге 4
- `prototype` — альтернатива, когда домен не ясен (вместо SPDD)
- `improve-codebase-architecture` — для рефакторинга существующего, не SPDD-кейс
- `to-issues` — после декомпозиции на истории (шаг 1) можно вынести в трекер
