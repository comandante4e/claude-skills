# SPDD — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `spdd`; логика и процесс — те же, что в [SKILL.md](SKILL.md): канвас REASONS, 6 шагов, гейт-ревью перед генерацией, правило «промпт — источник правды». Здесь только дельта механики под Codex.

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md` (корень проекта или `~/.codex/AGENTS.md`). Авто-триггера по `description` (как у Claude-скиллов) у Codex нет — поэтому скилл активируется одним из трёх способов:

1. **Правило в `AGENTS.md`** (рекомендуется) — Codex подхватывает его в каждой сессии и сам уходит в SPDD-процесс, когда видит формулировку «создать новую систему/сервис/MVP/модуль с нуля». Готовый блок ниже.
2. **Слэш-команда** — положи промпт в `~/.codex/prompts/spdd.md` (тело — методология из SKILL.md или ссылка на неё), вызывай `/spdd` в чате.
3. **Вручную** — скажи Codex «веди по SPDD» и сошлись на `skills/spdd/SKILL.md` + `REFERENCE.md`.

Минимальный блок для `AGENTS.md`:

```md
## SPDD (Structured Prompt-Driven Development)
Когда пользователь хочет построить НОВУЮ систему/сервис/MVP/модуль/крупную фичу с нуля
(«давай сделаем X», «спроектируем», «начнём новый проект») — веди по SPDD:
story → analysis → context → REASONS Canvas → generate (тесты, потом код) → unit tests.
Артефакты пиши в docs/spdd/<story-id>/ (story.md, analysis.md, context.md, reasons-canvas.md).
Канвас REASONS — quality gate ПЕРЕД генерацией кода: не пиши код без подтверждения пользователя.
Промпт — источник правды: расхождение промпта и кода → правь канвас первым, потом код. Не молча.
НЕ применять для hotfix, спайков, неясного домена, тривиальных правок.
Полная версия — skills/spdd/CODEX.md, шаблоны — skills/spdd/REFERENCE.md.
```

## Что меняется против Claude-версии

Скилл по своей природе **инструкционный**: это методология плюс шаблоны на диске, без субагентов, бандл-скриптов и MCP-инструментов. Поэтому переносится почти 1-в-1. Реальные дельты под Codex:

### 1. Артефакты на диске вместо «памяти агента»
SPDD и так строится вокруг файлов в `docs/spdd/<story-id>/` — это совпадает с тем, как Codex работает между сессиями: контекст не копится, входы читаются с диска. Если ведёшь шаги в **разных сессиях Codex** (например, `analysis` сегодня, `reasons-canvas` завтра), новая сессия читает уже записанные `story.md` / `analysis.md` / `context.md` и продолжает. Никакого спавна субагентов «на шаг» (как Task у Claude Code) у Codex нет и не требуется — артефакты и есть передача состояния. Для воспроизводимого/скриптового прогона шага можно использовать `codex exec "<инструкция>, входы читай из docs/spdd/<story-id>/"`.

### 2. Шаг 2 (Analysis): сканирование кода
В SKILL.md шаг Analysis опирается на Grep/Glob для скана существующего кода. В Codex это штатный поиск по репозиторию (`rg`/grep через шелл) — эквивалент есть, отдельной настройки не нужно. Никаких бандл-скриптов в скилле нет (`ls` показывает только `SKILL.md` и `REFERENCE.md`), запускать нечего.

### 3. Слэш-вызовы делегируемых скиллов
SKILL.md/REFERENCE.md ссылаются на смежные скиллы (`tdd`, `grill-me`, `grill-with-docs`, `prototype`, `improve-codebase-architecture`, `to-issues`). У Codex авто-вызова этих скиллов **нет**: на шаге 5 веди TDD-цикл (RED→GREEN→REFACTOR) сам, а жёсткое ревью канваса на шаге 4 проводи как явный диалог, а не вызовом `grill-me`. Если какой-то из этих скиллов нужен и его поведения нет под рукой — **не имитируй вызов**, проведи эквивалент вручную или скажи пользователю.

### 4. Гейты и стоп-сигналы — под GPT строже
- Канвас REASONS на шаге 4 — это quality gate: **не переходи к генерации кода без явного подтверждения пользователя**. Не перепрыгивай его «за компанию».
- Держи 6 шагов **явными и по чек-листу**, ревью на границах шагов.
- На шаге 5 — строго TDD: сначала acceptance-тесты по Requirements (красные), потом код по Operations шаг за шагом.
- Стоп-сигналы (реализация против Safeguards; entity не из Entities; тесты зелёные, но Requirements не выполнены; «просто поправь код») — при срабатывании останавливайся, правь канвас первым, сообщай пользователю.

## Ресурсные файлы

`REFERENCE.md` (шаблон REASONS Canvas, детали каждого шага, стоп-сигналы, связь со смежными скиллами) переносится **как есть** и читается Codex'ом с диска так же, как Claude — отдельной адаптации не требует.
