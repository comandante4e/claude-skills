# UI/UX Pro Max — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `ui-ux-pro-max`. Логика и процесс — те же, что в [SKILL.md](SKILL.md) (анализ требований → `--design-system` → доменные/стековые поиски → синтез → чек-лист перед сдачей). Здесь только дельта механики под Codex.

Суть скилла: это **не текстовые правила, а bundled-инструмент** — Python-CLI `scripts/search.py` (поверх `core.py` + `design_system.py`), который BM25-поиском роется по CSV-базе в `data/` (67 стилей, 96 палитр, 57 пар шрифтов, 99 UX-гайдлайнов, 25 типов чартов, 13 стеков). Значит, главный вопрос под Codex — **как этот скрипт запускается** (Codex его сам не дёрнет) и **как скилл вообще активируется** (авто-триггера по `description` нет).

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md` (корень проекта или `~/.codex/AGENTS.md`). **Авто-триггера по полю `description`, как у Claude-скиллов, у Codex НЕТ** — фронтматтер `SKILL.md` Codex игнорирует. Поэтому активируется одним из трёх способов:

1. **Правило в `AGENTS.md`** (рекомендуется) — Codex видит его в каждой сессии и знает, когда и как звать CLI.
2. **Кастомный промпт для слэш-вызова** — положи файл `~/.codex/prompts/ui-ux-pro-max.md` с тем же правилом; тогда в чате доступен вызов `/ui-ux-pro-max`.
3. **Вручную** — прямо просишь Codex «прогони ui-ux-pro-max по этому запросу».

Минимальный блок для `AGENTS.md` (укажи реальный путь к скиллу — например `~/.codex/skills/ui-ux-pro-max` или путь в репозитории):

```md
## UI/UX Pro Max (дизайн-интеллект)
Когда пользователь просит спроектировать / собрать / отревьюить UI или UX
(сайт, лендинг, дашборд, компонент, палитру, типографику, чарты), действуй так:
1. Извлеки из запроса: тип продукта, ключевые слова стиля, индустрию, стек
   (по умолчанию html-tailwind).
2. ОБЯЗАТЕЛЬНО запусти дизайн-систему:
   python3 <PATH>/ui-ux-pro-max/scripts/search.py "<тип> <индустрия> <стиль>" --design-system [-p "Имя проекта"]
3. По мере надобности — доменные и стековые поиски:
   python3 <PATH>/ui-ux-pro-max/scripts/search.py "<keyword>" --domain <domain>
   python3 <PATH>/ui-ux-pro-max/scripts/search.py "<keyword>" --stack html-tailwind
4. Синтезируй вывод и реализуй; перед сдачей пройди Pre-Delivery Checklist из SKILL.md.
Нужен Python 3 в PATH. CLI не запускается сам — вызывай его явно. Нет Python/файла —
остановись и сообщи, не выдумывай рекомендации «из головы».
Полная версия — <PATH>/ui-ux-pro-max/CODEX.md.
```

## Что меняется против Claude-версии

### 1. Bundled-скрипты: Codex их сам не запустит — вызывай явно (главный подводный камень)
В Claude Code модель сама дёргает `scripts/search.py` по инструкции из `SKILL.md`. **У Codex авто-запуска вспомогательных скриптов нет** — пока агенту/пользователю явно не сказано, он команду не выполнит. Поэтому:
- запуск всегда **полной командой** `python3 <PATH>/ui-ux-pro-max/scripts/search.py …` (а не «search.py …» из текущего каталога). `core.py`/`design_system.py` лежат рядом и импортируются скриптом сами, `DATA_DIR` вычисляется относительно файла — путь к `data/` ломаться не будет;
- для воспроизводимого/скриптового прогона удобен `codex exec "python3 <PATH>/.../search.py 'fintech crypto' --design-system"` (неинтерактивно); интерактивная сессия — для исследовательских итераций;
- если в `~/.codex/config.toml` стоит approval-policy с подтверждением команд — первый запуск Codex попросит апрув; разреши `python3` для этого пути, чтобы не дёргало на каждый поиск.

### 2. Зависимость от Python 3 (guard)
CLI требует Python 3 (стандартная библиотека: `csv`, `re`, `argparse` — внешних пакетов нет, BM25 реализован руками в `core.py`). Перед первым запуском проверь `python3 --version || python --version`; нет интерпретатора — поставь по инструкции из раздела Prerequisites в [SKILL.md](SKILL.md) и **сообщи пользователю**, а не подменяй поиск выдуманными палитрами/шрифтами. Это и есть правило «нет инструмента — не выдумывай, остановись».

### 3. Слэш-вызов: `/ui-ux-pro-max` через промпт-файл
В Claude Code скилл триггерится `description`-матчингом. В Codex эквивалент слэш-команды — файл `~/.codex/prompts/ui-ux-pro-max.md` (см. способ 2 выше), вызывается как `/ui-ux-pro-max`. Без него — правило в `AGENTS.md` или ручной запрос.

### 4. shadcn/ui MCP (опциональная интеграция из `description`)
`SKILL.md` упоминает интеграцию с shadcn/ui MCP для поиска компонентов и примеров. В Codex MCP-серверы настраиваются **пользователем** в `~/.codex/config.toml` (секция `[mcp_servers.…]`), а не подключаются скиллом. Если MCP shadcn не настроен — это не блокер: основной поток (`--design-system` + `--stack shadcn`) работает на локальных CSV без всякого MCP. Подключение MCP — отдельный шаг на стороне пользователя.

### 5. Чего здесь НЕТ
Скилл **не использует субагентов / свежий контекст на шаг** и **не опирается на git worktree / harness-изоляцию** — это однопроцессный CLI + синтез в одной сессии. Поэтому разделов про «отдельную сессию на команду» и «worktree на slice» (как в `void-workflow/CODEX.md`) здесь нет — они к этому скиллу неприменимы.

### 6. GPT-специфика
- Держись явного порядка из `SKILL.md`: сначала **обязательный** `--design-system`, потом доменные/стековые добивки, потом синтез — не перепрыгивай к коду мимо дизайн-системы.
- Перед сдачей UI пройди **Pre-Delivery Checklist** из `SKILL.md` по пунктам (иконки SVG не эмодзи, cursor-pointer, контраст light/dark, респонсив 375/768/1024/1440, a11y) — это явный чек-лист, под GPT работает надёжнее «на глазок».

## Ресурсные файлы и данные — переносятся как есть
`data/*.csv` (включая `data/stacks/*.csv`) и `scripts/*.py` переносятся **без изменений** и читаются Codex'ом так же — с диска, через тот же CLI. Persist-режим (`--persist [--page …]`) создаёт `design-system/<project>/MASTER.md` и `pages/*.md` в текущем рабочем каталоге — ровно как описано в [SKILL.md](SKILL.md); хелпер просто помни, что при иерархическом чтении страничный override бьёт MASTER.

## Доступные домены и стеки (как в SKILL.md)
Домены: `product`, `style`, `typography`, `color`, `landing`, `chart`, `ux`, `react`, `web`, `icons`.
Стеки: `html-tailwind` (DEFAULT), `react`, `nextjs`, `astro`, `vue`, `nuxtjs`, `nuxt-ui`, `svelte`, `swiftui`, `react-native`, `flutter`, `shadcn`, `jetpack-compose`.
Форматы вывода дизайн-системы: `-f ascii` (терминал, по умолчанию) / `-f markdown` (для документации).
