# Windows Shell — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `windows-shell`; логика и правила те же, что в [SKILL.md](SKILL.md), здесь только дельта под Codex.

Скилл почти полностью **инструкционный**: это набор правил про Windows-пути и шелл-команды (кавычить пути с бэкслешами, `/dev/null` вместо `>nul` в Git Bash, не пайпить `findstr`/`Where-Object` после `cd`, `-not` вместо `!` и `$var` через файлы `.ps1`). Сами правила переносятся под Codex **1-в-1** — это про синтаксис команд, который от агента не зависит. Меняется только одно: где скилл живёт/как триггерится и чем заменить раздел про «специализированные инструменты вместо Bash».

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md` (корень проекта или `~/.codex/AGENTS.md`). Авто-триггера по `description` (как у Claude-скиллов) у Codex нет — поэтому правила надо положить туда, где модель их гарантированно увидит, на Windows-машине:

- **`~/.codex/AGENTS.md`** — глобально для всех проектов на этой машине (рекомендуется: правило окружения, а не проекта).
- **кастомный промпт** `~/.codex/prompts/windows-shell.md` — для ручного вызова `/windows-shell` в чате, когда нужно подтянуть полный чек-лист.
- либо вставить блок в `AGENTS.md` конкретного репозитория.

Минимальный блок для `AGENTS.md` (Windows-машина):

```md
## Windows shell (Git Bash + PowerShell)
- Пути с бэкслешами всегда в кавычках: cd "C:\Repos\proj". Без кавычек бэкслеши теряются.
- В Git Bash редирект в null — только /dev/null (НЕ >nul: создаёт файл "nul" и засоряет git status).
- Не пайпить findstr / Where-Object после cd: имя каталога ловится как аргумент.
- powershell -Command после bash-cd теряет контекст — либо весь конвейер в PowerShell, либо без пайпа.
- В inline-PowerShell через bash: -not вместо "!", сложную логику с $-переменными выноси в .ps1.
- В Skills/конфигах — прямые слеши (scripts/helper.py), бэкслеши не использовать.
Полная версия — skills/windows-shell/CODEX.md.
```

Ресурсных файлов (`scripts/`, `data/`, отдельные `*.md`) у скилла нет — только `SKILL.md`. Так что переносить как есть нечего, кроме самих правил.

## Что меняется против Claude-версии

Скилл не использует субагентов, git-worktree, MCP или bundled-скрипты — соответствующих дельт здесь нет. Единственное реальное отличие:

### Раздел «Tools to Prefer Over Bash» — под Codex переосмысли
В SKILL.md есть таблица «используй `Read`/`Glob`/`Grep`/`Edit`/`Write` вместо `cat`/`find`/`grep`/`sed`/`echo>`». У Codex CLI **нет таких именованных инструментов** — он работает преимущественно через шелл. Поэтому:

- мотив таблицы под Codex меняется с «возьми спец-инструмент» на «не нарывайся на Windows-грабли шелла»: главная польза не в инструменте, а в том, что чтение/поиск/правка файлов в обход хрупких `cd ... | findstr` и inline-PowerShell **снимает ровно те проблемы**, ради которых написан скилл;
- если Codex настроен с MCP-файл-сервером (filesystem MCP в `~/.codex/config.toml`) — используй его операции чтения/правки вместо `cat`/`sed`, это и есть аналог «спец-инструментов»;
- если MCP нет — пользуйся аккуратными прямыми вызовами (`grep`/`rg` без пайпа после `cd`, путь через `git -C "C:\..."`), а не цепочками, которые ломает Windows.

То есть таблица остаётся валидным ориентиром «чем не стоит работать вслепую на Windows», но без привязки к конкретным Claude-инструментам.

### GPT-специфика и guard
- Перед каждой Windows-командой прогоняй **чек-лист** из конца SKILL.md явно (кавычки, `/dev/null`, нет пайпа после `cd`, `-not`, `.ps1`) — GPT-модели полезно проговаривать гейты, а не «срезать».
- Для воспроизводимых прогонов на Windows запускай команды через `codex exec "<...>"` — те же правила кавычек/`/dev/null` действуют и там.
- **Нет нужного шелла/инструмента — не выдумывай.** Если на машине нет Git Bash (а есть только PowerShell или cmd) либо отсутствует ожидаемый MCP-файл-сервер — остановись и сообщи пользователю, не имитируй `/dev/null`-семантику в среде, где её нет (в cmd/PowerShell нуль-устройство и редирект другие).
