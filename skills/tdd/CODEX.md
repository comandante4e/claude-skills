# Test-Driven Development — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `tdd`; логика и процесс те же, что в [SKILL.md](SKILL.md) — здесь только дельта под Codex.

Скилл **чисто инструкционный**: это методология (red-green-refactor, вертикальные срезы вместо горизонтальных, тесты на поведение через публичный интерфейс, deep modules, мокабельность на границах системы). Нет субагентов, нет git worktree-изоляции, нет bundled-скриптов, нет MCP-инструментов. Поэтому под Codex он **переносится 1-в-1, поведение не меняется** — отличается только то, ГДЕ он живёт и КАК активируется.

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md` (корень проекта или `~/.codex/AGENTS.md`). Авто-триггера по полю `description` (как у Claude-скиллов) у Codex **нет** — фронтматтер из `SKILL.md` Codex не использует. Три рабочих варианта:

1. **Правило в `AGENTS.md`** (рекомендуется) — короткий блок ниже в `AGENTS.md` проекта (или глобально в `~/.codex/AGENTS.md`). Тогда Codex держит TDD-дисциплину в контексте на всех задачах репозитория.
2. **Кастомный промпт для слэш-вызова `/tdd`** — положи тело `SKILL.md` (и при желании ресурсные `.md`) в `~/.codex/prompts/tdd.md`. В чате `/tdd` подставит этот текст как инструкцию. Аналог `/tdd` из Claude Code.
3. **Вручную** — просто скорми содержимое `SKILL.md` в начале сессии, когда строишь фичу или чинишь баг через тесты.

Минимальный блок для `AGENTS.md`:

```md
## TDD (разработка через тесты)
Когда строишь фичу или чинишь баг через тесты — иди строго red → green → refactor,
ВЕРТИКАЛЬНЫМИ срезами: один тест → минимальный код, чтобы он прошёл → следующий тест.
НЕ писать все тесты пачкой, потом весь код (горизонтальный срез даёт «crap tests»).
Тесты проверяют ПОВЕДЕНИЕ через публичный интерфейс, не детали реализации; не мокать
свой код — мок только на границах системы (внешние API, БД, время, ФС). Перед кодом —
согласовать с пользователем интерфейс и какие поведения тестировать (нельзя протестировать
всё). Рефакторить только на GREEN, никогда на RED. Полная версия — skills/tdd/CODEX.md
и skills/tdd/SKILL.md.
```

## Что меняется против Claude-версии

Поведенческих дельт нет — красно-зелёный цикл, чек-листы фаз (Planning → Tracer Bullet → Incremental Loop → Refactor) и правило «один тест за раз» работают у GPT-Codex буквально так же. Применимы только две поправки на среду:

- **Расположение и триггер** (см. выше): `AGENTS.md` / `~/.codex/prompts/tdd.md` / ручная вставка вместо авто-активации по `description`.
- **GPT-специфика — держи гейты явными.** Не перепрыгивай шаги чек-листа из `SKILL.md`: согласование интерфейса и набора поведений ДО кода; `RED` (тест падает) перед `GREEN`; рефактор только после того, как все тесты зелёные. Для воспроизводимого прогона тестов в цикле используй `codex exec "<команда тестов>"` (неинтерактивно) — это удобный способ зафиксировать, что текущий тест реально падает на RED и проходит на GREEN, не накапливая контекст сессии.

Чего тут **нет** (и придумывать не надо): субагентов/свежего контекста на шаг, worktree-параллели, bundled-скриптов, MCP. Все эти секции из harness-heavy адаптаций к `tdd` неприменимы.

## Ресурсные файлы

Пять справочных файлов рядом со скиллом переносятся **как есть** и читаются Codex'ом с диска так же, как Claude'ом:

- [tests.md](tests.md) — примеры хороших/плохих тестов
- [mocking.md](mocking.md) — когда и где мокать, dependency injection, SDK-style интерфейсы
- [deep-modules.md](deep-modules.md) — маленький интерфейс + глубокая реализация
- [interface-design.md](interface-design.md) — проектирование интерфейсов под тестируемость
- [refactoring.md](refactoring.md) — кандидаты на рефакторинг после цикла

Если активируешь через `~/.codex/prompts/tdd.md`, можешь либо вклеить нужные из них в тело промпта, либо оставить ссылками — Codex дочитает их с диска по пути, когда дойдёт до соответствующей фазы.
