# Handoff — версия для Codex CLI (GPT под капотом)

Это Codex-адаптация скилла `handoff`; логика/процесс те же, что в [SKILL.md](SKILL.md) — здесь только дельта под Codex. Скилл чисто инструкционный: ни субагентов, ни worktree, ни MCP, ни bundled-скриптов. Поведение переносится **1-в-1**; меняется только где он живёт, как триггерится и пара деталей механики под Codex.

## Куда положить и как активировать

Codex автоматически читает `AGENTS.md` (корень проекта или `~/.codex/AGENTS.md`). Авто-триггера по `description` (как у Claude-скиллов) у Codex **нет** — нужно либо правило в `AGENTS.md`, либо кастомный промпт-файл.

Три способа активировать этот скилл под Codex:
- **Кастомный промпт `/handoff`** — положи тело скилла в `~/.codex/prompts/handoff.md`, тогда в чате доступен слэш-вызов `/handoff <для чего следующая сессия>`. Аргумент после слэша играет роль `argument-hint` из фронтматтера (описание фокуса следующей сессии).
- **Правило в `AGENTS.md`** — глобально (`~/.codex/AGENTS.md`) или в корне проекта; срабатывает на фразу-триггер вроде «сделай handoff» / «компактни диалог для следующей сессии».
- **Вручную** — просто вставить инструкцию в чат текущей сессии.

Минимальный блок для `AGENTS.md`:

```md
## Handoff (передача контекста следующей сессии)
Когда пользователь говорит «сделай handoff» (или вызывает /handoff) — собери
handoff-документ по текущему диалогу, чтобы свежая сессия Codex продолжила работу.
Путь файла получи через `mktemp -t handoff-XXXXXX.md`, прочитай файл перед записью.
Не дублируй то, что уже есть в PRD/планах/ADR/issue/коммитах/диффах — ссылайся
на них путём или URL. Предложи, какие скиллы пригодятся следующей сессии.
Аргумент после /handoff — это фокус следующей сессии, подгони документ под него.
Полная версия — skills/handoff/CODEX.md.
```

## Что меняется против Claude-версии

1. **«Свежий агент, который подхватит работу» → новая сессия Codex / `codex exec`.** У Codex нет встроенного спавна субагентов. «Следующий агент» из SKILL.md — это просто **отдельный запуск Codex** (новая интерактивная сессия или `codex exec "<...>"`), который **читает handoff-файл с диска**. Поэтому путь к файлу должен быть явным: получаешь его через `mktemp` и сообщаешь пользователю, чтобы он скормил его новой сессии (например, `codex exec "Прочитай <путь к handoff.md> и продолжи работу"`). Историю текущего диалога новая сессия не наследует — весь нужный контекст должен жить в файле.

2. **`mktemp` — обычная shell-команда, запусти явно.** У Codex нет авто-исполнения вспомогательной обвязки: сам вызови `mktemp -t handoff-XXXXXX.md`, забери путь, **прочитай файл перед записью** (под Codex это не делается харнессом автоматически — read-before-write выполняешь руками), затем запиши документ.

3. **Аргумент фокуса.** В Claude Code это `argument-hint` слэш-команды. Под Codex тот же аргумент приходит либо как текст после `/handoff` (если оформлен промпт-файлом), либо как часть запроса в чате — трактуй его как описание задачи следующей сессии и подгоняй документ под неё.

4. **«Предложи скиллы для следующей сессии».** Скилла Claude под Codex может не быть в том же виде — предлагай эквиваленты (правило в `AGENTS.md` / промпт в `~/.codex/prompts/`), а не Claude-специфичные слэш-команды. Нет подходящей механики — так и скажи, не выдумывай.

Ресурсных файлов (REFERENCE.md, *-FORMAT.md и т.п.) у скилла нет — переносить нечего.
